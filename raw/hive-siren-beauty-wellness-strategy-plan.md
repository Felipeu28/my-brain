PART IV
Design Direction &amp; Visual Language
The new Siren website must feel like entering a sanctuary — not a landing page. Every visual, typographic, and interaction choice must communicate luxury, intention, and warmth. We build ON the brand the client loves, not away from it.
Design Principles
Atmospheric Luxury: The site must feel like the physical space itself — warm, refined, slightly mysterious. Think candlelit calm meets scientific precision.
Editorial Over Transactional: Services are not menu items — they are curated experiences. Design each service like a magazine spread.
Intentional Whitespace: Premium brands breathe. Generous spacing signals confidence and elevates every element.
Photography-Led: The brand's visual identity lives in its imagery. The site should feel like a curated lookbook.
Motion with Restraint: Subtle parallax, fade-in reveals, and smooth scroll transitions add depth without gimmickry.
Mobile-First Luxury: Over 70% of visitors will arrive on mobile. The mobile experience must match the desktop's richness.

PART VI
SEO &amp; AI Search Optimization Strategy
In 2026, ranking in Google and appearing in AI-generated answers (ChatGPT, Perplexity, Google AI Overviews, Claude) requires a two-layer strategy: traditional SEO for the algorithm and Answer Engine Optimization (AEO) for the AI systems.
Technical SEO Foundation
Schema Markup Implementation
LocalBusiness schema (MedSpa subtype) — NAP, hours, geo-coordinates, services
Service schema for each individual service page
FAQPage schema on FAQ page and individual service pages
Person schema for provider profile — builds E-E-A-T signals
Review / AggregateRating schema — pulls Google stars into search results
BreadcrumbList schema — improves site navigation in SERPs
Article schema on all blog posts
All schema implemented as JSON-LD (not microdata) for clean implementation
Core Web Vitals &amp; Performance
LCP (Largest Contentful Paint) target: under 2.5 seconds
CLS (Cumulative Layout Shift) target: under 0.1
FID / INP target: under 200ms
Image optimization: WebP format, lazy loading, proper dimensions
Font loading: font-display: swap, preload key fonts
No render-blocking scripts — defer all non-critical JavaScript
CDN implementation for all static assets
On-Page SEO Architecture
Each page has a unique, keyword-rich title tag (50-60 characters)
Unique meta descriptions for every page (150-160 characters) — include location + service + CTA
H1 → H2 → H3 hierarchy strictly maintained on every page
Internal linking strategy: service pages link to each other, blog posts link to service pages
Image alt text with descriptive, keyword-aware language on every image
Canonical tags to prevent duplicate content across location pages
XML sitemap submitted to Google Search Console
Robots.txt properly configured
Local SEO Strategy
Google Business Profile Optimization
Complete all 150+ GBP fields — name, address, phone, website, hours, services, attributes
Add all services with descriptions and pricing ranges
Weekly GBP posts — promotions, tips, seasonal offers
Photo cadence: 2-4 new photos per week (interior, team, treatments, results)
Q&amp;A section: seed 10-15 common questions with detailed answers
Review response strategy: respond to every review within 24 hours
Enable GBP booking link — connect to online booking system
Local Citation Building
Ensure NAP (Name/Address/Phone) is 100% consistent across: Google, Yelp, Facebook, Fresha, Allē, Healthgrades, ZocDoc, RealSelf
Submit to local Buda/Hays County business directories
Austin Woman Magazine (they featured Saving Face) — pursue editorial coverage
Community Impact newspaper — already featured Siren, maintain relationship for future coverage
Chamber of Commerce membership — backlink and local authority
Answer Engine Optimization (AEO) for AI Search
AEO is the practice of structuring content so AI systems (ChatGPT, Perplexity, Google AI Overviews, Claude) cite your business when answering user queries. This is the fastest-growing search channel in 2026.
AEO Content Principles
Answer-first format: lead every page and blog post with a direct, complete answer to the primary question
Conversational query targeting: write FAQ content that mirrors how people ask questions out loud ('How long does botox last in Buda TX?', 'What's the best medspa near Kyle Texas?')
Entity consistency: same business name, address, phone, and description across ALL web properties — AI systems verify entity data across multiple sources
E-E-A-T (Experience, Expertise, Authoritativeness, Trustworthiness): provider credentials, certifications, years of experience prominently displayed on every service page
Define specialized terms: AI systems favor sites that explain industry terminology clearly (what is Sculptra, what is CO2 Cool Peel)
Structured data everywhere: FAQ schema is the single most direct pipeline to AI answer boxes

'How much does botox cost in Austin/Buda area' — pricing transparency page
'Difference between botox and filler' — educational blog posts

PART VIII
Google Stitch 2.0 + Claude: Implementation Workflow
The Google Stitch → Claude pipeline (launched March 2026) represents the most powerful AI-native web development workflow available today. Stitch handles the design generation; Claude handles the code execution. Together they eliminate the traditional design-to-development handoff bottleneck.
The Workflow
Step 1: Design Brief → Stitch
Use Google Stitch 2.0 to generate the visual design system: color palette, typography, component library, layout grids, and page wireframes from our design brief
Step 2: Vibe Design Mode
Use Stitch's 'vibe design' workflow — describe business objectives and emotional intent (luxury, warmth, sanctuary) rather than pixel specs
Step 3: Design.md Export
Export Stitch's Design.md file — a machine-readable design system document containing all tokens, components, and layout rules
Step 4: Stitch MCP Connection
Connect Stitch to Claude via the Stitch MCP server — Claude reads the design data directly through the MCP connection (layout, colors, components)
Step 5: Claude Code Generation
Claude reads the Design.md and generates production-ready Next.js + Tailwind components that precisely match the Stitch design
Step 6: Iteration Loop
Designer refines in Stitch → Claude regenerates updated components → repeat until perfect
Step 7: Content Integration
Claude integrates Sanity CMS schema and connects components to live content
Step 8: SEO &amp; Schema Layer
Claude implements all JSON-LD schema markup, meta tag system, and sitemap generation
Step 9: Performance Audit
Claude runs Lighthouse audits and iterates until Core Web Vitals targets are met
Step 10: Launch Preparation
Claude sets up Vercel deployment, environment variables, and final QA checklist
Claude Sonnet vs. Opus: Which Model for This Project?
RECOMMENDATION: Use Claude Sonnet 4.5+ as the primary development model, with Claude Opus 4 reserved for architectural planning, complex problem-solving, and final quality review passes.
Claude Sonnet — Primary Development Model
Optimal for: component generation, iterative code writing, CSS/Tailwind styling, schema markup
Speed advantage: faster response times for high-iteration tasks like refining components
Quality: excellent for well-defined, structured tasks with clear design specs from Stitch
Cost-efficiency: lower cost per iteration enables more refinement cycles within budget
Stitch integration: Sonnet's context window handles full Design.md files and component libraries
Claude Opus — Strategic &amp; Complex Tasks
Use for: initial project architecture decisions, complex SEO strategy implementation, Sanity schema design, debugging difficult edge cases
Use for: writing long-form service page content that requires nuanced brand voice
Use for: final review and quality assurance pass across the full codebase
Reserve for: any task where architectural correctness matters more than iteration speed
Optimal Prompting Strategy for Claude
Always start with the Design.md context from Stitch — Claude performs dramatically better with design system constraints
Provide brand voice examples in every content generation prompt
Use system prompts to establish: brand name, color hex values, font names, tone guidelines, SEO requirements
Request components in isolation first, then composite page layouts — smaller, verifiable units
Ask Claude to include schema markup, accessibility attributes (aria-labels), and SEO meta simultaneously — not as afterthoughts
Use Claude to generate the Sanity schema first — this creates a structural contract the rest of the codebase can follow
For performance: ask Claude to explain its rendering strategy for each component — SSG vs SSR vs client-side
